## 2. 슬라이드별 상세 초안 (Slide-by-Slide Draft)

### Slide 1: 프로젝트 정의 및 범위
- **Title**: 프로젝트 정의: 데이터 주도형 웹 플랫폼
- **Key Bullet Points**:
  - 단순한 정적 팀 소개 웹사이트가 아닌 통합 데이터 플랫폼.
  - Next.js(FE), 로컬 데이터 파이프라인, Supabase(서빙), 내부 Admin/Ops 도구로 구성.
  - **핵심 대원칙**: 퍼블릭 페이지는 런타임에 외부를 스크래핑하지 않고, 검증 완료된 '서빙 데이터'만 읽음.
- **Speaker Notes**:
  "안녕하십니까. 본 프레젠테이션에서는 NZU 프로젝트의 기술적 기반을 소개합니다. 이 시스템은 단순 브로셔 사이트가 아닙니다. 자동화된 데이터 수집과 검증 파이프라인 위에서 동작하는 '데이터 주도형 스포츠/팀 정보 플랫폼'으로 이해하셔야 합니다. 가장 중요한 원칙은 라이브 웹사이트의 안정성을 위해 런타임 스크래핑을 배제하고, 철저히 검증된 데이터만 서빙한다는 점입니다."

### Slide 2: 단순한 홈페이지가 아닌 이유
- **Title**: 왜 단순한 홈페이지를 넘어섰는가?
- **Key Bullet Points**:
  - 원천 데이터(Source of truth)와 서빙 데이터(Serving data)의 철저한 분리.
  - 자동화된 수집과 인간의 수동 보정(Admin)이 공존하는 운영 파이프라인 내장.
  - 미래의 확장을 고려한 영구적 플레이어 식별자(Stable Join Key) 체계.
- **Speaker Notes**:
  "이 프로젝트가 일반적인 홈페이지와 구별되는 지점은 내부 구조에 있습니다. 데이터의 원천과 실제 웹에서 보여주는 서빙 레이어를 극단적으로 분리했습니다. 또한, 시스템이 모든 것을 완벽히 자동화할 수 없다는 현실을 반영하여, 내부 관리자 도구를 통한 수동 보정 프로세스를 시스템의 정식 기능으로 설계했습니다."

### Slide 3: 전체 시스템 아키텍처
- **Title**: 3계층 시스템 아키텍처 (3-Layer Architecture)
- **Key Bullet Points**:
  - **Layer A (수집 및 정규화)**: 로컬 스크립트를 통한 데이터 수집, 검증, 리포트 생성.
  - **Layer B (서빙 레이어)**: 승인된 데이터만 Supabase(`players`, `matches` 등)로 푸시.
  - **Layer C (웹 및 운영)**: Next.js 앱을 통한 퍼블릭 렌더링 및 Admin/Ops 인터페이스 제공.
- **Speaker Notes**:
  "시스템은 세 가지 레이어로 구성됩니다. 첫째, 백그라운드에서 데이터를 수집하고 정규화하는 레이어. 둘째, 검증을 통과한 데이터만 보관하고 제공하는 Supabase 서빙 레이어. 셋째, 실제 유저와 관리자가 접속하는 Next.js 레이어입니다. 이 구조 덕분에 프론트엔드와 데이터 파이프라인이 독립적으로 안전하게 동작합니다."

### Slide 4: 데이터 흐름 - 수집부터 서빙까지
- **Title**: 무결성을 보장하는 데이터 흐름 (Data Flow)
- **Key Bullet Points**:
  - 수집 ➔ 정규화 ➔ 로컬 검증 ➔ 승인 ➔ Supabase 동기화 ➔ 퍼블릭 서빙.
  - **운영 원칙**: 수집과 동시에 라이브 서버로 직행하지 않음 (`Collect & Validate First`).
  - 부정확한 데이터 유입을 차단하는 승인 및 동기화 분리 구조.
- **Speaker Notes**:
  "우리의 데이터 파이프라인의 철학은 '선 수집 및 검증, 후 동기화'입니다. 스크립트가 데이터를 긁어와서 곧바로 라이브 DB에 넣는 것이 아니라, 중간에 로컬 검증과 승인 단계를 거칩니다. 이 격리 구조가 라이브 서비스의 품질과 안정성을 보장합니다."

### Slide 5: 메타데이터 및 식별자 구조
- **Title**: 이중 구조의 메타데이터 및 식별자 (Metadata & Identity)
- **Key Bullet Points**:
  - **Master Identity DB**: 프로젝트와 무관하게 변하지 않는 고유 플레이어 식별자 보관.
  - **Project/Team Roster DB**: 특정 팀 소속, 티어, 종족 등 프로젝트 종속적인 상태 정보 보관.
  - 향후 확장 및 타 프로젝트 재사용을 위한 견고한 Stable Join Key (`entity_id`, `eloboard_id`) 활용.
- **Speaker Notes**:
  "데이터 모델에서 가장 공을 들인 부분입니다. '플레이어 존재 자체(Identity)'와 '현재 소속/상태(Roster)'를 분리했습니다. 마스터 DB는 변하지 않는 기준점 역할을 하며, 로스터 DB는 이번 NZU 프로젝트만의 운영 상태를 가집니다. 이는 훗날 다른 대회가 열리더라도 시스템을 쉽게 재사용할 수 있는 기반이 됩니다."

### Slide 6: 퍼블릭 웹사이트 서빙 모델
- **Title**: 빠르고 안전한 퍼블릭 서빙 (Public Service Architecture)
- **Key Bullet Points**:
  - Next.js App Router와 React 19 아키텍처 적용 (일부 ISR-style Revalidation 활용).
  - 로컬 메타데이터 파일을 직접 파싱하거나 외부를 스크래핑하지 않고, 전용 서비스(`player-service.ts`)를 통해 Supabase만 조회.
  - 홈, 티어별 목록, 플레이어 상세 페이지, 배틀 그리드 등으로 데이터 재가공 표출.
- **Speaker Notes**:
  "퍼블릭 웹사이트, 즉 일반 유저들이 보는 화면은 매우 가볍고 안전하게 동작합니다. 프론트엔드에서는 복잡한 데이터 파싱이나 외부 스크래핑을 절대 수행하지 않습니다. 오직 Supabase에 예쁘게 정리된 데이터만을 `player-service`를 통해 읽어와 쾌적한 UX를 제공합니다."

### Slide 7: 관리자 및 운영 모델
- **Title**: 운영을 위한 내장 관리자 도구 (Admin & Ops)
- **Key Bullet Points**:
  - 단순 열람을 넘어, 파이프라인 스냅샷 확인 및 데이터 수동 보정 기능 내장.
  - 로스터 이동, 티어 변경, 예외 처리(Exclusion/Resume), 강제 덮어쓰기 로드(Lock) 등 지원.
  - 수동 개입 내역을 추적하고 자동화와 충돌하지 않도록 조율하는 실용적 패턴.
- **Speaker Notes**:
  "이 프로젝트가 돋보이는 것은 '운영 관리자 페이지'가 구색 맞추기가 아니라 실제 문제를 해결하도록 짜여 있다는 점입니다. 데이터가 틀렸을 때 개발자가 DB를 직접 수정하는 것이 아니라, 관리자 UI에서 플레이어를 이동시키고 강제 고정을 걸 수 있습니다. 실무적인 생산성을 크게 높여주는 영역입니다."

### Slide 8: 자동화 및 GitHub Actions 워크플로우
- **Title**: GitHub Actions 기반 스케줄 자동화
- **Key Bullet Points**:
  - 일일 단위 파이프라인 자동 실행, 회귀 테스트(Regression Test), 검증 알람 로직 탑재.
  - 상태 리포트를 생성하고 Discord 웹훅을 통한 운영진 알림 전송.
  - 자동화 중에도 Supabase 동기화는 Opt-in 방식 및 Secret으로 보호.
- **Speaker Notes**:
  "시스템 자동화는 GitHub Actions가 담당합니다. 정해진 시간에 의존성을 설치하고 파이프라인을 돌리며, 그 결과를 디스코드로 쏴줍니다. 흥미로운 점은 자동화 워크플로우 내에서도 DB 동기화는 안전장치(Opt-in)가 마련되어 있어, 치명적 오류가 운영 DB로 확산되는 것을 방지한다는 것입니다."

### Slide 9: 프로덕션 성숙도: 완성 vs 과도기 영역
- **Title**: 프로덕션 성숙도 분석 (Mature vs Transitional)
- **Key Bullet Points**:
  - **성숙 방향성 (Mature)**: 데이터 수집/검증 파이프라인, 메타데이터 구조, GitHub Actions 스케줄링, Supabase 서빙 레이어, 관리자 수동 보정 워크플로우 등 시스템의 핵심 아키텍처는 이미 안정적인 방향을 갖추고 있음.
  - **과도기 영역 (Transitional)**: 라이브 플랫폼(현재 부분 목업 상태), 매치/엔트리 보드(UI 프로토타입 위주), 파일 기반 로컬 Analytics, 단순화된 Admin 인증.
- **Speaker Notes**:
  "현재 시스템의 아키텍처는 비교적 성숙한 상태입니다. 데이터 수집 분리부터 관리자 워크플로우까지의 뼈대는 상용 수준의 견고함을 지향합니다. 다만, 사용자에게 보여지는 일부 영역(라이브 연동, 매치 엔트리 로직 등)과 통계 분석 기능은 아직 완성된 모듈이 아닌 전환기적 상태에 머물러 있음을 솔직히 공유해 드립니다."

### Slide 10: 현재 기술적 병목 영역
- **Title**: 시스템의 현재 병목 및 한계 (Current Technical Bottlenecks)
- **Key Bullet Points**:
  - **서빙 레이어 준비도 불균형**: `players` 데이터는 안정적으로 연동되었으나, `matches`, `eloboard_matches` 등 매치 기반 서빙은 아직 비어 있음.
  - **페이지별 레디니스 격차**: 플레이어 기반 뷰(`/tier`, `/player`)는 완성도가 높으나, 매치/라이브 주도 뷰(`/`, `/live`, `/match`)는 여전히 목업 중심의 프로토타입 UI.
  - **파편화된 통계 분석 아키텍처**: 일부 앱은 파일(CSV)에, 일부는 DB(Supabase)에 의존하는 하이브리드 구조라 운영 복잡성 증가.
  - **관리자 권한 및 환경 불안정성**: 단일 키(`ADMIN_ACCESS_KEY`) 기반 인증의 한계 및 로컬 빌드 시(spawn EPERM) 기록된 환경 재현성 문제.
- **Speaker Notes**:
  "현재 프로젝트가 겪고 있는 명확한 병목이자 기술적 한계점들입니다. 데이터 구조나 아키텍처 비전은 탄탄하게 잡혀 있으나, 이를 100% 퍼블릭 레벨까지 끌어올리는 연결 고리가 부족합니다. 플레이어 정보는 잘 나오지만 매치 실황 등 핵심적 흐름은 여전히 UI 프로토타입에 의존하며, 데이터 분석과 로컬 환경 안정성에서도 기술 부채가 남아 있습니다."

### Slide 11: 외부 전문가 기대 수준 및 협업 범위
- **Title**: 외부 전문가에게 기대하는 기술 협업 (Expected Expert Help)
- **Key Bullet Points**:
  - **데이터 서빙 갭(Gap) 해소**: 비어 있는 매치 워크플로우 구축 및 서빙 데이터 격차를 체계적으로 메우는 설계. 
  - **데이터 드리븐 워크플로우 전환**: 뛰어난 UI 중심 프로토타입(라이브, 매치/엔트리 뷰)을 실제 백엔드 데이터에 연동.
  - **분석 아키텍처 확립**: 현재의 하이브리드 분석 계층을 Supabase 중심으로 일원화할지 분리할지에 대한 아키텍처 방향성 확정.
  - **단계적 론칭(Staged Launch) 컨설팅**: 플레이어 우선 공개 ➔ 매치 중심 추후 오픈 등 명확한 기술 로드맵 제시 및 환경 정상화.
- **Speaker Notes**:
  "우리가 외부 파트너 및 전문가분들께 기대하는 핵심 가치는 바로 이 지점입니다. 단순히 빈칸을 채우는 코딩 외주가 아니라, UI 프로토타입과 수집 파이프라인 사이에 끊어진 워크플로우를 데이터 드리븐으로 융합하는 설계가 필요합니다. 아키텍처의 불안 요소들을 잡아주고 단계적으로 론칭할 수 있는 튼튼한 다리가 되어주시길 기대합니다."

### Slide 12: 향후 기술적 우선순위
- **Title**: 다음 기술 성장 과제 (Next Priorities)
- **Key Bullet Points**:
  - **Live Data Connect**: 목업(Mock)을 넘어 실제 SOOP TV 등 외부 파트너 스트리밍 API 연동.
  - **Match Orchestration**: 프로토타입 UI에 머문 매치/엔트리 보드를 실제 데이터 모델과 연결되는 엔진으로 격상.
  - **Analytics Architecture**: 외부 전문가 가이드를 수용하여, 분석 계층을 파일 기반으로 계속 유지할지 Supabase 중심으로 옮길지 아키텍처 결정을 명확히 수립.
  - **Admin Auth Hardening**: 외부 및 참가팀 확장 대비 강력한 관리자/유저 권한 인증 체계 재구축.
- **Speaker Notes**:
  "마지막으로 향후 프로젝트 우선순위입니다. 외부 전문가와 함께 실시간 방송 API의 라이브 연동, 매치 워크플로우 백엔드 처리, 그리고 통계 분석 아키텍처 결정이라는 굵직한 과제를 순차적으로 돌파해 나갈 것입니다. 이 과정이 완료되면, 본 프로젝트는 명실상부한 완성형 스포츠 데이터 운영 시스템으로서 가치를 증명할 것입니다."

## 3. 강력하게 통제되어야 할 5가지 핵심 메시지 (Five Critical Messages)

아래 5가지 메시지는 프레젠테이션 과정이나 외부 소통 시 **절대 왜곡되거나 과장되어서는 안 되는** 핵심 기술 원칙입니다:

1. **퍼블릭 페이지는 절대 런타임 스크래핑을 하지 않는다.** (오직 미리 검증되고 동기화된 안전한 Serving 데이터만 읽는다.)
2. **로컬 메타데이터가 식별자의 진짜 원천(Source of Truth)이다.** (Supabase는 이를 빠르고 효율적으로 전달하기 위한 '캐시/서빙' 레이어일 뿐이다.)
3. **완벽한 자동화라는 환상 대신, 예측 가능한 수동 보정이 시스템의 일부로 설계되어 있다.** (Admin 툴은 부가 기능이 아니라 데이터 무결성의 핵심 부품이다.)
4. **수집과 라이브 동기화는 물리적으로 분리되어 있다.** (수집하자마자 퍼블릭으로 쏘지 않으며, 반드시 검증/승인이라는 방화벽을 거친다.)
5. **라이브 연동과 매치 생성 기능은 아직 완성된 프로덕션 모듈이 아니다.** (화려한 UI로 구현되어 있으나 현재는 프로토타입/과도기적 기능임을 투명하게 밝혀야 합니다. 이를 과장하지 않고 인정하는 것이 전문가 협업의 시작입니다.)
