# 데이터 파이프라인 재정의 (2026-07-19 시작)

목표: **"엘로보드에서 가져온다 → 알맞게 다듬는다 → 호사가에 보여준다"**를
정확하고, 심플하고, 간편하게. 다른 사이트에도 응용 가능한 자산으로.

사용자(산박)와 문답으로 한 항목씩 확정한다. 이 문서가 유일한 설계 기준이며,
이후 모든 파이프라인 정리 작업은 이 문서에 맞춘다.

**설계 6항목 전부 확정 완료 (2026-07-21).** 남은 것은 구현: §3의 선수 대장 통합
2일 작업. 요약:
1. 수집 규칙 = 책갈피(증분)
2. 선수 식별 = 새 번호는 새 기록, 이름 fallback 금지, 종족변경 승계는 legacy_entity_ids
3. 예외는 대장 하나 · 3종류(임시/고정/제외)
4. 이상 시 **수집은 완주, 서빙 반영만 보류**
5. 진실은 **Supabase 테이블 하나**
6. 보고는 **성공/실패 + 판단할 것만**

---

## 1. 무엇을 어떻게 가져오나 — ✅ 확정 (2026-07-19)

### 어디서, 무엇을

엘로보드에서 3가지:

1. **팀 목록** — 새 팀 자동 감지
2. **팀별 선수 명단** — 이름, 티어, 종족, 엘로보드 번호 (팀당 1페이지)
3. **선수별 경기 기록** — 날짜, 상대, 승/패. 2025-01-01 이후.
   **선수 개인 페이지에서만** 수집 가능(엘로보드에 전체 경기 목록 페이지 없음 — 사용자 확인).
   한 경기가 양쪽 선수에게서 2번 수집되는 것은 허용 비용.

숲 ID·방송 상태·프로필 사진은 엘로보드에 없음 → 별도 경로(숲 API + 수동 입력).

### 언제

매일 밤 1회 (06:10 KST, GitHub Actions). 저장소 공개(public) 유지 시 무료 무제한.
비공개 전환 시 월 2,000분 한도 초과(현 사용량 ~2,200분/월)이므로 실행 시간 단축 선행 필요.

### 어떻게 — 책갈피 3줄 규칙

> 1. **평소: 모든 선수, 책갈피(증분)로 읽는다** — 지난번 지점부터 새 경기만.
>    근거: 과거 기록은 바뀌지 않는다 (사용자 확인).
> 2. **수상할 때: 그 선수만 전체 다시 읽는다** — 이상 감지 시에만 정독.
> 3. **보험: 매일 소수씩 돌아가며 전체 검증** — 하루 ~10명 순환, 한 달에 전원 1회.
>    조용한 삭제·정정을 한 달 내 반드시 발견.

추가 절약: 14일 이상 경기 없는 선수는 확인 건너뜀 (현행 유지).

### 현행과의 차이 = 수리 목록

| # | 현행 | 수리 방향 |
|---|------|-----------|
| R1 | 활발한 선수(매일 확인 대상)만 `--no-cache` 전체 재수집 | 책갈피(증분)로 전환 |
| R2 | 책갈피(`tmp/.cache/roster_report_cache.json`)가 GitHub Actions 캐시에만 존재 — 증발 시 조용히 전체 재수집 | 영구 저장소(R2 등)에 보관 |
| R3 | 순환 전체 검증 없음 | 신규 추가 (하루 ~10명) |

수리 시점: 임시 교정 자동 해제(b576307)의 첫 실전 가동(2026-07-20 아침) 확인 후.
하룻밤에 변화는 하나씩.

## 2. 선수를 무엇으로 식별하나 — ✅ 확정 (2026-07-19)

> 1. **선수의 신분증 = 엘로보드 번호(entity_id).** 이름이 아니다.
> 2. **이름만 같다고 절대 자동으로 같은 사람 취급하지 않는다.** (동명이인 위험.
>    현행 코드의 이름 fallback 조인은 제거 방향.)
> 3. **재등록 의심 시 시스템이 묻는다.** 같은 팀에서 옛 번호 소멸 + 같은 이름의
>    새 번호 등장 → 디스코드로 "이어붙이기 후보" 보고. **확정은 사용자.**
> 4. **"새 번호 = 새 기록."** 재등록(종족 전환 포함)이면 무조건 새 출발.
>    옛 번호의 경기 기록은 버린다 — 합산하지 않는다.

### 새 출발 원칙의 이유 (사용자 결정)

종족 전환(예: 세월 테란→프로토스) 시 옛 종족 시절 기록이 새 프로필에 섞이면
시청자가 혼란("프로토스인데 왜 테란 기록이?"). 스타 판 감각으로 사실상 다른 선수.
조건부 규칙(종족 바뀌면 새 출발 / 안 바뀌면 합산)은 복잡도만 늘리므로 무조건 통일.

### 이어붙이기(successor mapping)의 재정의

기록을 잇는 게 아니라 **운영 정보만** 승계:

| 승계함 | 승계하지 않음 |
|--------|---------------|
| 숲 ID, 프로필 사진 | ❌ 경기 기록 |
| "사라진 선수" 알림 방지 (은퇴 아님, 전환임) | ❌ 승률/통산 통계 |
| 팀 합계 감소를 "예상된 감소"로 처리 (2026-07-19 HM -268 오탐 방지) | |

참고: 2026-07-19 HM -268 가드 오탐은 이 원칙 관점에서 "데이터는 맞았고
가드만 이어붙이기를 모르고 놀란 것". 가드가 successor mapping을 인지하도록 수리 필요.

## 3. 자동 vs 수동의 우선순위 — ✅ 확정 (2026-07-19)

> 1. **엘로보드 값이 기본값이다.** 손대지 않으면 엘로보드 그대로.
> 2. **예외는 장부 하나에만 적는다.** 이유·날짜와 함께.
> 3. **예외의 종류는 3가지뿐:**
>    - **임시** — 엘로보드가 틀린 동안만. 일치해지면 자동 해제 (`b576307` 구현 완료)
>    - **고정** — 엘로보드에 없어서 수동 관리 (YB 팀, 숲 ID 등)
>    - **제외** — 수집/표시 안 함 (외부인, 은퇴)

### 현황 진단 (2026-07-19 정밀 인벤토리)

예외 저장소가 12곳으로 퇴적됨. 원인 3가지:
- 의도된 이중화 (로컬 파일 + Supabase `roster_admin_corrections` — 관리자 수정이 커밋 없이 반영되기 위함. **유지**)
- 사고 처방전이 그대로 파일이 됨 (3/21 잠금, 3/21 제외, 4/9 숲ID, 5/21 상대선수...)
- 같은 사실 복제 (별명 "프발=이광용"이 4개 파일에, 숲 ID가 최대 4곳에)

죽은 파일 2개: `identity_alias_exceptions.v1.json`(0건), `pipeline_collection_resumes.v1.json`(미존재).

### 통합 설계: 선수 대장 (player ledger)

**파일 하나, 선수(entity_id)당 한 줄.** Supabase `roster_admin_corrections`가 이미
교정+제외+재수집+숲ID를 한 테이블에 통합한 성공 전례 — 이 구조를 확장.

선수 행: 교정(팀/티어/종족, 임시|고정) · 숲 ID · 표시 이름 · 다른 표기(별명) ·
이어붙이기(legacy_entity_ids, 기록 승계 없음 — 2번 원칙) · 제외/재수집 · 무기록 허용 · 메모/날짜.
별도 섹션 하나: **외부인 이름 목록** (우리 선수가 아닌 상대 이름, 현 126건).

흡수 대상: roster_manual_overrides, pipeline_collection_exclusions/resumes,
player_display_aliases, opponent_identity_aliases, opponent_identity_review_decisions,
identity_alias_exceptions(삭제), alert_rules 내 zero_record allowlist,
soop_channel_mappings(선수 매핑 부분), team_role_overrides(역할 순서).
팀 단위 allowlist(fa)는 알림 규칙에 잔류.

### 전환 계획 — 2일 (전·후 결과 비교로 검증)

검증 원칙: 옛 코드/새 코드로 각각 시험 실행(report-only) → **결과물이 동일함을 diff로 증명**
+ 전체 계약 테스트. 밤 실행은 최종 확인 역할.

- **1일차 ✅ 완료 (2026-07-21)**: 죽은 파일 삭제(`identity_alias_exceptions`),
  대장 생성(`data/metadata/player_ledger.v1.json`) + 공유 로더(`scripts/tools/lib/player-ledger.js`),
  상대선수 목록(`opponent_identity_review_decisions`)·상대 별명(`opponent_identity_aliases`) 흡수 후 삭제.
  리더 8곳 + validator 전환. 검증: 옛/새 report-only diff 동일 + `test:player-ledger` + 파이프라인 테스트 전체.
  **수집 제외(`pipeline_collection_exclusions`)는 이연** — 관리자 화면(roster-admin-store·admin API·ops-review)도
  읽어 "파이프라인 전용"이 아니라 관리자 영역과 함께 처리해야 한다(당초 설계 가정과 다름).
- **2일차**: 수동 교정·숲 ID·표시 별명 흡수 — **사이트 화면이 직접 읽는 영역이라 분리**
  (lib/player-serving-metadata.ts). 수집 제외도 여기서 함께. 빌드 + 실제 페이지 확인 후 배포. 옛 파일 삭제로 완료.

## 4. 뭘 "이상"으로 보고 멈추나 — ✅ 확정 (2026-07-21)

> **수집은 항상 끝낸다. 이상은 "사이트 반영(서빙 동기화)"만 멈춘다.**

경기 기록 수집과 사이트 반영을 분리한다. 이상이 감지돼도 수집은 끝까지 돌려
데이터를 모으고(손실 없음), 의심스러우면 **서빙 동기화만** 건너뛴다.

- **근거**: 2026-07-19 HM 팀 레이팅 -268 경보(세월 identity 교체 부작용, 오탐)가
  수집 전체를 중단시켜 하루치를 날렸다. 수집은 되돌릴 수 없지만(그날 방송은 그날뿐),
  서빙 반영은 다음 날 다시 하면 된다. 막아야 할 것은 **틀린 데이터가 사이트에 나가는 것**
  이지 수집이 아니다.
- **현행과의 차이**: 지금은 `blocking_severities=[critical, high]` 경보가 수집 단계
  (collect_chunked)까지 실패시킨다. 앞으로 blocking은 `supabase_push`(서빙 동기화)만
  막고, 수집·warehouse 적재는 항상 완주한다.
- 경보 등급은 그대로 쓰되 의미를 재정의: blocking = "서빙 반영 보류" (수집 중단 아님).

## 5. 어디에 실어 보내나 (진실은 한 곳) — ✅ 확정 (2026-07-21)

> **Supabase 테이블 하나가 진짜 장부다.** (`roster_admin_corrections` 확장)

예외가 지금 9곳(로컬 파일 8 + 원격 테이블 1)에 흩어져 있다. 이것을 §3의
**선수 대장** 하나로 합치고, 그 대장을 **Supabase 테이블**로 둔다.

- **왜 Supabase인가**: `roster_admin_corrections`가 이미 교정+제외+숲ID를 한 테이블에
  통합한 성공 전례다. 관리자가 **커밋 없이 수정**할 수 있는 게 핵심 장점 —
  이걸 잃으면 안 된다. git 파일로 하면 모든 예외 변경이 커밋·배포를 거쳐야 한다.
- **로컬 파일의 운명**: 흡수 후 삭제. 단, 파이프라인 스크립트가 원격을 못 읽는
  환경(오프라인 재현 등)을 위해 **읽기 전용 스냅샷**을 git에 남길지는 §3 전환 때 결정.
- **주의**: 어제 예실 교정이 로컬(fa)과 원격(bgm)에서 달랐던 혼란이 바로 이중화 탓.
  단일화하면 이 종류의 어긋남이 원천적으로 불가능해진다.

## 6. 관측: 보고와 알림 — ✅ 확정 (2026-07-21)

> **성공/실패 + 사람이 판단할 것만.** 조용한 날은 조용하게.

매일 아침 디스코드 보고는 두 가지만 담는다: ①파이프라인 성공/실패 ②사람의 판단이
필요한 항목(확인 필요·해제 대기·이름충돌 수집제외·기록 0건). 팀별 수치·델타 나열은
뺀다 — 매일 비슷해서 중요한 게 묻힌다.

- **철학**: 경보가 **신호**가 되게 한다. 매일 뜨는 것은 경보가 아니다.
  2026-07-20에 "확인 필요 27건"이 매일 떠도 아무도 안 봤다(0건으로 정리함).
- **`npm run pipeline:now`와 같은 관점**: 이 명령이 이미 "판단할 것만" 보여준다.
  아침 보고는 이걸 디스코드로 밀어주는 것이라고 보면 된다.
- **조용한 날**: 성공 + 판단할 것 없음 → 한 줄 요약. 상세는 링크로.
- 남은 선택(구현 때): 판단할 것이 0건인 날은 아예 안 보낼지, 한 줄이라도 보낼지.
  → "잘 돌고 있다"는 확인은 필요하므로 **한 줄은 보낸다** 쪽으로 기운다(구현 때 확정).
