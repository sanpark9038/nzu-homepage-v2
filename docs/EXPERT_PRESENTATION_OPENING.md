# Expert Presentation Opening

## 2-3 Minute Opening Script

안녕하세요. 오늘은 NZU 홈페이지 프로젝트를 단순한 팀 소개 사이트가 아니라, `데이터 수집 파이프라인`, `서빙 계층`, `프론트엔드`, 그리고 `운영 도구`가 결합된 구조로 설명드리려고 합니다.

이 프로젝트의 출발점은 단순히 예쁜 웹사이트를 만드는 것이 아니었습니다. 실제 선수 정보, 전적, 티어, 팀 소속, 그리고 운영 중 발생하는 예외 상황까지 안정적으로 관리할 수 있는 구조를 만들고 싶었습니다.

그래서 현재 시스템은 크게 세 층으로 나뉘어 있습니다.

첫 번째는 로컬 데이터 파이프라인입니다. 여기서 외부 데이터를 수집하고, 정규화하고, 검증하고, 리포트를 만듭니다.

두 번째는 Supabase 서빙 계층입니다. 중요한 점은 이 레이어가 원천 데이터가 아니라, 검증을 통과한 데이터를 퍼블릭 웹사이트에 전달하는 역할이라는 것입니다.

세 번째는 Next.js 기반의 웹과 관리자 도구입니다. 일반 사용자는 공개 페이지를 통해 선수/티어/팀 정보를 보게 되고, 운영자는 관리자 도구를 통해 수동 보정이나 파이프라인 상태 확인을 하게 됩니다.

이 프로젝트에서 가장 중요하게 생각한 원칙은 하나입니다.

`퍼블릭 페이지는 런타임에 외부를 직접 스크래핑하지 않는다.`

즉, 수집과 서빙을 분리해서 사이트 안정성을 지키는 것이 핵심 설계 철학입니다.

현재 기준으로는, 파이프라인과 메타데이터 구조, 그리고 관리자 운영 흐름은 비교적 방향이 잘 잡혀 있습니다. 반면 라이브 데이터 연동, 매치/엔트리 보드, 분석 계층 정리, 관리자 인증 고도화는 아직 과도기적입니다.

그래서 오늘 이 발표의 목적은 “우리가 무엇을 만들었는지 자랑하는 것”보다, 현재 구조를 정확히 공유하고, 어디에서 전문가의 도움이 가장 의미 있는지를 명확히 보여드리는 데 있습니다.

핵심적으로는 다음 질문에 대해 의견을 얻고 싶습니다.

1. 현재의 파이프라인-서빙 분리 구조를 어떻게 더 단단하게 고도화할지
2. 비어 있는 match/eloboard serving gap을 어떤 방식으로 메울지
3. 라이브/매치 기능을 프로토타입에서 실제 데이터 워크플로우로 어떻게 전환할지
4. 분석 계층을 파일 기반과 DB 기반 중 어떤 방향으로 정리하는 것이 맞는지

오늘은 이 관점으로 시스템을 설명드리겠습니다.
