# NZU Homepage UI Supplementary PPT Draft

보조 PPT는 시스템의 핵심 백엔드 아키텍처가 실제 사용자 인터페이스(UI)로 어떻게 노출되는지 시각적으로 보여주기 위한 4장 분량의 짧은 자료입니다. 각 슬라이드는 실제 화면 캡처(스크린샷)를 중심으로 구성됩니다.

---

## Slide 1: 퍼블릭 서빙 화면 (Public Serving Screens)
- **Title**: 검증된 데이터의 최종 종착지: 퍼블릭 서비스 화면
- **Key Message**: 런타임 스크래핑 없이, 파이프라인을 거쳐 정제된 '안전한 서빙 데이터'만 유저에게 노출됩니다.
- **Visuals (Screenshots)**:
  1. `[스크린샷 삽입: /tier 화면]` - 티어 및 종족별로 정리된 안정적인 플레이어 인벤토리.
  2. `[스크린샷 삽입: /player/[id] 화면]` - 개별 선수의 상세 정보 및 전적 통계를 보여주는 상세 뷰.
- **Speaker Notes**:
  "첫 번째로 보실 화면들은 일반 유저들이 접근하는 퍼블릭 영역입니다. 이 화면들의 가장 큰 특징은 무거운 연산이나 외부 크롤링 없이, 오직 Supabase에 예쁘게 적제된 데이터를 빠르게 읽어와 뿌려준다는 점입니다. 현재 가장 성숙하고 안정적으로 구동되는 영역입니다."

---

## Slide 2: 고도화된 내부 운영 도구 (Private Admin & Ops Screens)
- **Title**: 시스템의 빈틈을 메우는 인간 개입: 어드민/운영 도구
- **Key Message**: 완전 자동화의 한계를 인정하고, 데이터 무결성을 보장하기 위해 설계된 강력한 수동 개입 인터페이스입니다.
- **Visuals (Screenshots)**:
  1. `[스크린샷 삽입: /admin/ops 화면]` - 현재 수집 상황, 알람 발생 여부, 파이프라인 수동 실행 버튼 등 운영 현황 스냅샷.
  2. `[스크린샷 삽입: /admin/roster 화면]` - 선수의 소속 팀 이동, 티어 조작, 데이터 예외 처리(Exclusion)를 위한 로스터 조작 UI.
- **Speaker Notes**:
  "단순 홈페이지라면 이런 화면이 필요 없습니다. 하지만 우리는 데이터 플랫폼이기 때문에, 파이프라인이 놓친 오류를 운영진이 직접 교정할 수 있는 어드민 화면을 핵심 기능으로 구축했습니다. 수집 에러를 확인하고 로스터를 강제 조정하는 일련의 워크플로우가 이 프라이빗 뷰에서 일어납니다."

---

## Slide 3: 프로토타입 및 과도기적 기능 (Transitional Screens)
- **Title**: 성장 중인 라이브 & 매치 인프라: 과도기 영역
- **Key Message**: UI 프로토타입은 훌륭하게 구현되었으나, 백엔드의 실제 데이터 엔진 연동(Live API, Match Orchestration)이 필요한 영역입니다.
- **Visuals (Screenshots)**:
  1. `[스크린샷 삽입: /live 화면]` - UI는 라이브 플랫폼 형태를 갖췄으나, 현재 목업(Mock) 데이터가 일부 포함된 뷰.
  2. `[스크린샷 삽입: /match 혹은 /entry 화면]` - 향후 대진표 작성 및 경기 관리를 위한 시각적 셸(Shell) 형태의 엔트리 보드.
- **Speaker Notes**:
  "기술적으로 가장 병목이자, 향후 전문가와 함께 풀어가야 할 화면들입니다. 겉보기엔 그럴싸한 실황 중계 UI와 대진표 화면이 준비되어 있지만, 백엔드에는 아직 이를 뒷받침할 라이브 API와 매치 생성 엔진이 완벽히 연결되지 않았습니다. 우리가 조언을 얻고자 하는 메인 타겟 영역입니다."

---

## Slide 4: UI와 메인 아키텍처의 연결 구조 (UI to Architecture Mapping)
- **Title**: 화면 뒤에 숨겨진 3계층 코어 아키텍처 매핑
- **Key Message**: 
  위에서 살펴본 화면들은 메인 아키텍처의 각 레이어(수집 ➔ 보정 ➔ 서빙)와 정확히 1:1로 맞닿아 있습니다.
- **Visuals (Diagram / Summary)**:
  - `Layer A (Ingestion & Validation)` ➔ 👁️ **모니터링**: `/admin/ops`
  - `Manual Correction (Admin)` ➔ 🔧 **수동 개입**: `/admin/roster`
  - `Layer B (Supabase Serving)` ➔ 🚀 **웹 서빙**: `/tier`, `/player/[id]`
  - `Future Work (API Connection)` ➔ 🚧 **협업 대기**: `/live`, `/match`
- **Speaker Notes**:
  "이 모든 화면은 복잡한 아키텍처 맵과 그대로 연결됩니다. 수집 파이프라인의 결과는 ops 화면에서 감시되고, 관리자의 보정은 roster 화면에서 이뤄집니다. 최종 승인된 데이터는 tier 화면에 뿌려지고, 남은 과제인 API 연동은 live와 match 화면을 채울 것입니다. 이렇게 화면만 보셔도 우리 시스템의 데이터 흐름 전체를 직관적으로 파악하실 수 있습니다."
